## Current Telemetry

**What is captured:**
- `session_id` — unique Claude Code session identifier
- `project` — project name (last path segment of cwd)
- `cwd` — absolute working directory at session end
- `ts_end` — Unix timestamp when session ended
- `total_tokens` — cumulative tokens for entire session
- `phases` — list of phase transitions, each with:
  - `ts` — ISO timestamp of transition
  - `phase` — `PENDING`, `IMPLEMENTING`, or `COMPLETED`
  - `task_id` — plan path or spec description (nullable)
  - `tokens_cumulative` — running token count at transition moment

**Where stored:** `~/.claude/devflow/telemetry/sessions.jsonl` (append-only, one JSON object per line, deduplicated by session_id)

**How phases are detected:**
1. Explicit: writes to `active-spec.json` (Write tool or Bash)
2. Inferred fallback: IMPLEMENTING ← first source file Write/Edit after PENDING; COMPLETED ← last successful test-runner result after IMPLEMENTING

**Current data state (5 sessions, all 2026-03-31):**
- Projects: `startup` (3 sessions), `momease` (1 session), + 1 partial
- **Critical observation:** All 5 sessions have only `PENDING` recorded — zero sessions reached `IMPLEMENTING` or `COMPLETED`. The understand/build ratio (the core metric) has never been computed in production. This means the explicit active-spec.json write workflow defined in `devflow-spec-driven-dev` is not being followed consistently in practice, and the passive inference (first source-file write) is also not triggering. The telemetry infrastructure is intact but yielding no signal.

## Current LLM-as-Judge

**None exists.**

There is no LLM-based evaluation infrastructure in devflow. The closest approximation is:
- `pr-review-toolkit:review-pr` — an external superpowers agent that does code review, but it runs as a skill/agent invoked manually or at DONE, not as a hook that captures structured output
- `devflow-behavior-contract` — a structured contract format but evaluated by the user, not by an LLM judge

No structured criteria, no output schema, no scoring, no automated quality signal is captured from any LLM evaluation.

## Gaps Identified

### Missing Telemetry Signals

1. **Tool call frequency per spec cycle** — no measurement of how many Write/Edit/Read/Bash/Grep calls are made per PENDING→COMPLETED cycle. This is a direct proxy for agent efficiency and context navigation cost.

2. **Retry/failure rate before success** — no tracking of how many failed test runs occur before the first successful one. High retry count signals poor initial code quality or misaligned test assumptions.

3. **Compaction frequency per project** — PreCompact fires but nothing records how often it fires per project over time. Frequent compaction on the same project = architectural context scatter.

4. **Hook latency** — file_checker runs dart analyze / eslint / cargo check; no measurement of how long hooks take. Slow hooks degrade the interactive experience invisibly.

5. **Skill invocation tracking** — no signal for which skills are invoked, how often, and in which session contexts. Cannot measure whether model-routing or agent-orchestration skills are actually being used.

6. **tdd_enforcer follow-through rate** — the hook warns about missing tests, but there is no telemetry tracking whether a test file is subsequently created in the same session. The advisory is unverifiable.

7. **Review gate pass/fail rate** — pr-review-toolkit:review-pr runs as a skill but its output is not structured or captured. No signal on how often it finds issues, what categories, or whether they get fixed.

8. **USD cost estimation** — tokens are counted but no conversion to dollar cost. With multi-project telemetry, cost per feature is a natural derived metric.

9. **File length trend** — no tracking of whether files that trigger warnings in one session get split in a subsequent session. The enforcement is point-in-time, not longitudinal.

10. **Context window headroom at task start** — no measurement of how full the context is when `/spec` is invoked. High initial token count = less room for the task = higher compaction risk.

### Missing Enforcement Mechanisms

1. **No coverage gate** — `tdd_enforcer` is advisory-only. Nothing blocks a commit or push when a new impl file has no test. The quality signal fires but enforcement is absent.

2. **No commit message validation** — no hook enforces conventional commit format or any message structure. Atomic commit discipline (stated in CLAUDE.md) is not machine-enforced.

3. **No branch naming validation** — no hook enforces branch naming conventions.

4. **No secrets detection** — no PreToolUse gate on Write/Edit that scans for credentials, API keys, or `.env` leakage before writing. This is a material security gap for a tool used across multiple projects.

5. **No circular dependency detection** — no import graph analysis for Python or TypeScript projects.

6. **No duplicate code detection** — copy-paste is not flagged.

7. **No dependency vulnerability gate** — no npm audit / cargo audit / pip-audit check on pre-push.

8. **Toolchain gap: Python** — `file_checker.py` handles Node.js, Flutter, Go, Rust, Maven — but Python has no formatter/linter enforcement (black, ruff, flake8). Given that devflow itself is Python, this is an ironic gap.

9. **Toolchain gap: Python pre-push gate** — `pre_push_gate.py` also has no Python runner (pytest, mypy).

### Missing Eval Infrastructure

1. **No LLM-as-judge** — no structured evaluation of: code correctness, test quality, behavior contract adherence, architectural consistency, or output format compliance. Quality is gated by human review only.

2. **No benchmark task set** — no repeatable canonical tasks to measure devflow's effectiveness over time. Cannot answer "does version X of devflow produce better outputs than version Y?"

3. **No context efficiency scoring** — the understand/build ratio is measured (when it works) but is not automatically surfaced as architectural debt in the issue tracker. A ratio consistently >0.5 should auto-generate a tech debt draft.

4. **No regression detection** — no mechanism to compare the current session's ratio against the project's historical baseline. First sign of architectural degradation is invisible until it becomes severe.

5. **No skill effectiveness measurement** — no signal on whether invoking `devflow-behavior-contract` actually reduces regression rate compared to sessions without it.

### Technical Debt Items

1. **`discovery_scan.py` has zero test coverage** — this is the most complex hook: toolchain detection, issue tracker detection, design system detection, symlink management, project profile persistence. None of it is tested. Any regression is silent. Priority: HIGH.

2. **README badge says 197 tests but 200 pass** — stale badge. Minor but signals the README is not maintained in sync with the codebase.

3. **Telemetry has zero production signal** — 5 sessions, 0 complete phase cycles. The core value proposition of devflow (understand/build ratio as architectural quality proxy) has never produced a data point. Root cause: explicit active-spec.json write workflow is not being followed; passive inference is also not triggering. Either the session JSONL path resolution is failing, or the spec workflow is bypassed. Requires investigation.

4. **`pre_compact._find_active_spec()` uses markdown heuristic** — scans `~/.claude/plans/*.md` files looking for the string "IMPLEMENTING" or "in_progress". This is fragile: any plan document that _discusses_ an IMPLEMENTING phase will produce a false positive. It should read `active-spec.json` directly (which `spec_stop_guard` already does correctly).

5. **State directory pollution from tests** — unit tests create state dirs like `hook-session-99`, `real-session-id`, `s1` in `~/.claude/devflow/state/` and don't clean them up. These are not test artifacts in a tmp_path — they persist on the real filesystem after test runs.

6. **Learned skill symlink architecture has a race condition on parallel sessions** — documented in CLAUDE.md as a known issue, but no mitigation is enforced at the hook level. The `learned_skills_auto_inject: false` config option is a manual workaround, not a solution.

7. **Context window constant is hardcoded** — `CONTEXT_WINDOW_TOKENS = 200_000` in `_util.py`. When a model with a different context window is used (Haiku, Opus with extended context), the percentage calculations in `context_monitor.py` will be wrong. No mechanism to read the actual window size from the session.

8. **`task_telemetry.py` has no Python toolchain support** — `_SOURCE_EXTS` includes `.py` but `_TEST_RUNNER_RE` has no pattern for `pytest` — wait, actually it does (`\bpytest\b`). False alarm. But `_is_test_success` only checks for certain output patterns; pytest's concise output format (`N passed`) may not always be captured.

9. **`/pause` is implemented as raw Bash in the command file** — the command embeds a Python one-liner as a bash heredoc. If the path resolution fails silently, the user gets "No active spec to pause" when there is one. There's no hook (e.g., a dedicated `pause.py`) with tests verifying the pause logic.

10. **`/sync` has no implementation** — the `sync.md` command describes what it does conceptually but contains no actual implementation. It's a stub that asks Claude to perform a discovery scan via natural language. Unlike `/spec`, `/learn`, and `/pause` which have Python hooks backing them, `/sync` is entirely LLM-instruction-following with no deterministic enforcement.

## Bug fixes

### Fix: task_telemetry sessions stuck at PENDING (`2026-03-31`)

**Symptom:** 5 production sessions had only a `PENDING` phase in `sessions.jsonl`. No session had ever reached `IMPLEMENTING` or `COMPLETED`. The understand/build ratio — the core metric of devflow — had never produced a data point.

**Root cause:** The Claude Code Stop hook fires **after every assistant response turn**, not just at true session end. The deduplication logic in `main()` used `if session_id in existing_ids: return 0`, which:

1. After turn 1 (where PENDING is written to `active-spec.json` via Write tool): stop hook fires, parses the JSONL, finds only PENDING, writes the partial record to `sessions.jsonl`.
2. After turn 2+ (where IMPLEMENTING and COMPLETED are written): stop hook fires again, but `session_id` is already in `existing_ids` → returns 0, discarding the now-complete data.

This was confirmed by running `parse_session()` directly against the stuck sessions' JSONLs — all returned full PENDING + IMPLEMENTING + COMPLETED phase lists, proving the data was always there; it was just never written after the first partial recording.

**Fix:** Changed the deduplication logic from _skip if exists_ to _replace if exists_. When the stop hook fires for a session already recorded, the existing record is dropped from the in-memory list and replaced with the fresh, more-complete parse. The file is rewritten atomically via `write_text`.

**Files changed:** `hooks/task_telemetry.py` — `main()` function, lines 305–333.

**Tests added (3):**
- `test_main_lifecycle_pending_then_completed`: simulates the exact production scenario — stop hook fires after turn 1 (PENDING only), then again after turn N (full cycle). Asserts final record contains all 3 phases.
- `test_main_upsert_replaces_stale_pending_record`: pre-populates sessions.jsonl with a PENDING-only record, then runs with a complete JSONL. Asserts the record is replaced (not duplicated).
- `test_main_completed_record_has_non_null_key_fields`: asserts a completed record has `total_tokens > 0` (context_tokens proxy), IMPLEMENTING + COMPLETED phases (phase_durations derivable), and non-null `task_id` (task_category proxy).

**Test suite:** 200 → 203 tests, all passing.
